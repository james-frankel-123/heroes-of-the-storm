# Repository Guidelines

## Project Structure & Module Organization
- `src/app`: Next.js App Router entries, layouts, and route-specific UI.
- `src/components`: Reusable UI primitives and patterns (TSX + Tailwind).
- `src/lib`: Domain logic, data access (Drizzle queries), and helpers; unit tests live in `src/lib/__tests__` and `src/lib/draft/__tests__`.
- `public`: Static assets served as-is.
- `drizzle`: SQL migrations generated by Drizzle.
- `sync`: TypeScript scripts for data synchronization and endpoint checks.
- `training`: Python experimentation and GPU scripts; keep isolated from web build.

## Build, Test, and Development Commands
- `npm run dev` тАФ Start Next.js locally on port 3002.
- `npm run build` / `npm start` тАФ Production build and serve.
- `npm run lint` тАФ ESLint (Next preset) for TS/TSX and config files.
- `npm run type-check` тАФ Strict TypeScript validation.
- `npm test` / `npm run test:watch` тАФ Vitest unit suite.
- `npm run db:generate | db:migrate | db:push | db:studio` тАФ Drizzle schema/migration workflows.
- `npm run sync` тАФ Execute data sync tasks in `sync/`.

## Coding Style & Naming Conventions
- TypeScript-first; prefer async/await and explicit return types for shared helpers.
- 2-space indentation; keep imports sorted logically (framework, libs, local).
- Components and types use `PascalCase`; hooks/utilities `camelCase`; constants `SCREAMING_SNAKE_CASE` when shared.
- Styling via Tailwind; favor `clsx` and `class-variance-authority` for conditional classes; avoid inline hex colors when a token exists.
- Keep comments minimal and purposeful (clarify intent or invariants).

## Testing Guidelines
- Framework: Vitest; file pattern `*.test.ts`/`*.test.tsx` colocated under `src/lib`.
- Prefer pure functions for logic and cover edge cases (empty data, invalid tiers, extreme counts).
- Run `npm test` before PRs; include relevant test output in the PR description.

## Security & Configuration Tips
- Store secrets (e.g., `DATABASE_URL`, analytics keys) in `.env.local`; never commit them.
- For DB changes, update `drizzle.config.ts`, run `npm run db:generate`, and check generated SQL into `drizzle/`.
- When touching `sync/` or `training/`, document any external requirements (GPU, CUDA, API keys) in your PR.

## Commit & Pull Request Guidelines
- Commit messages follow concise, imperative subjects (e.g., `Optimize MCTS rollout`, `Fix tooltip alignment`) as seen in repo history.
- Keep commits scoped; avoid drive-by formatting unrelated to the change.
- PRs should include: summary of changes, linked issues, screenshots or clip for UI tweaks, test results (`npm test`, lint/type-check), and notes on schema or config impacts.
- Request review early for risky or cross-cutting changes; add TODOs with owner/context when deferring work.
